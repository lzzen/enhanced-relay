# 评审结论与补充建议

本文件是对 `requirements / architecture / observability-security / testing / roadmap / deployment / review-checklist` 七份规划的独立评审结论，并给出必须补充的决策与内容。评审目标：在正式编码前消除战略性歧义，确保“AI 自动实现 + 自动验收”闭环可以真正落地，避免人工陷入重复验证。

## 0. 总体评价

规划质量高，已覆盖多数资深团队才会考虑的问题：SSRF 与重定向绕过、图片解压炸弹、生图“模糊失败”幂等、计费快照冻结重算、慢请求分段 Trace、透明代理紧急降级、有界 worker pool。以下评审聚焦**尚未闭合的风险与缺口**，不重复已经写好的内容。

严重程度约定：

- `P0`：不解决会导致架构返工或线上事故，编码前必须定稿。
- `P1`：对应阶段开工前必须定稿。
- `P2`：规模化前定稿。

---

## 1. 战略路线（已定稿）

> **决策（2026-07-06）：采用「重构」路线。** 在 new-api 代码库基础上演进出增强能力，而非新建独立旁路网关。同时，本次重构必须引入**事件机制 + 钩子机制**，为后续插件式开发奠定基础。详见 [`docs/plugin-architecture.md`](plugin-architecture.md)。

因此下列由此决策派生的后续项：

- **拓扑重解释（重要）**：原架构文档的 Ingress/Egress “双代理环绕现有网关”是旁路方案的产物。在重构方案下，**不存在被环绕的第三方“现有网关”**——Ingress 与 Egress 退化为**同一个重构网关内部的两个流水线阶段**（面向客户端的入口阶段、面向上游的出口阶段）。原始请求 vs 实际上游请求的“双观察点”仍然保留，但由同一进程的两个 Hook 阶段采集，而非两个独立进程。架构文档第 1 节需据此改写。
- **能力去重**：new-api 已有的路由/渠道分组/额度计费/用户体系/relay 适配器/React 控制台**予以复用并重构**，不再重复新建。新设计中的“路由引擎/计费引擎/控制台”应表述为对现有能力的重构增强。
- **适配器即插件**：new-api 现有 `relay/channel` 的 `Adaptor` 注册表已是事实上的插件机制，本次将其**泛化**为统一插件/钩子体系的一种插件类型（上游适配器）。

仍需在实现前细化的重构子选项（记入 ADR）：

| 子选项 | 含义 | 建议 |
| --- | --- | --- |
| B. 原地演进 | 在 new-api 现有包内逐步重构出流水线与钩子 | 适合控制平面（用户/渠道/前端） |
| C. 数据平面抽取重写 | 保留用户/渠道/前端数据，重写 relay 数据平面为新流水线 | 适合 relay 热路径，便于达成资源隔离与快照不可变 |

> 建议：**控制平面原地演进 + 数据平面（relay）抽取重写**的混合式，热路径干净、存量资产不丢。

## 2. P0 缺失：迁移与并存策略

无论选 A/B/C，都缺少迁移计划。当前文档默认“绿地新建”，但现实是有存量：

- [ ] `P0` 存量渠道、令牌、用户、账本如何迁移或桥接？现有客户端令牌是否需要零改动继续可用？
- [ ] `P0` new-api 默认使用 SQLite；架构文档写“生产不建议 SQLite”。迁移到 PostgreSQL/MySQL 的路径、停机窗口、回滚方案？
- [ ] `P0` 数据库 schema 管理工具未定（goose / atlas / GORM AutoMigrate）。AutoMigrate 在生产多实例滚动升级下不安全，需要显式迁移工具与版本表。
- [ ] `P1` 并存期流量切分：按模型、按渠道、按用户灰度从旧链路切到新链路的开关粒度。

补充建议：新增 **Phase M（迁移与并存）** 或在 Phase 0 内明确“影子读取 + 不写、可零风险回退”的接入方式。

## 3. P0 缺失：内部跳段的信任与鉴权

架构规定“真实上游目标只能来自服务端 Provider 配置，禁止客户端伪造”，方向正确，但**跳段之间的信任链未定义**：

- [ ] `P0` 现有网关 → Egress 的 `/egress/provider-17` 路由如何鉴权？若无内部鉴权，任何能访问内网的实体都能借 Egress 发起对上游的调用（内部 SSRF / 开放代理）。建议内部 mTLS 或签名短期令牌 + Provider 白名单双重校验。
- [ ] `P0` 代理循环检测：`review-checklist` 已列出该问题但无机制。建议注入并校验 `Via` / 自定义 hop 头，超过跳数即拒绝。
- [ ] `P1` API Key（上游供应商密钥）存储与轮换：属于最高敏感度密钥，文档未涉及 KMS/加密存储/轮换/最小权限。

## 4. P1 性能目标与可观测成本的内在矛盾

- **“透明代理额外 P95 < 5 ms” 与 “Trace 覆盖率 100% + Prometheus 全量”存在张力。** 全量 OTel span 导出、mutex 计时、结构化事件在 5ms 预算内需要严格设计：span 异步导出、计时用单调时钟且合并、指标用无锁计数、事件走 ring buffer 异步落盘。建议：
  - [ ] `P1` 明确 Trace 采样策略：**100% 采集分段耗时（本地聚合）** 与 **跨系统 OTel 导出采样（如尾部采样保留慢/错请求）** 分离，避免把导出成本算进热路径。
  - [ ] `P1` 定义 5ms 的**测量方法学**（在什么负载、什么机型、扣除网络后如何测量、基线对照组是纯 `httputil.ReverseProxy`）。否则该指标无法自动验收。
- [ ] `P1` 明确“无规则快速通道”是否连 Body 都不缓冲（真正的流式透传），这是达成 5ms 的关键前提。

## 5. P1 缺失：配置、密钥与多租户

- [ ] `P1` 配置快照的**来源、schema、版本化、分发与失效通知**只在架构里一句带过。需要：配置 schema 校验、原子切换、灰度下发、坏配置自动回滚、启动时 fail-fast。
- [ ] `P1` 密钥管理：上游 API Key、OSS 凭证、调试快照加密密钥、内部 mTLS 证书的存放与轮换。
- [ ] `P1` 多租户模型：是否有租户维度？限流、并发、配额、审计策略是否按租户隔离？文档出现“租户级并发限制”但无租户模型定义。

## 7. 测试体系评审（对应用户强诉求）

现有 `testing.md` 是**优秀的“测什么”清单**，但用户明确要求“**AI 自动测试验收，不要让人工陷入其中**”。当前缺少“**谁来跑、如何自动判定通过、如何防止 AI 写假测试**”这一层。为此单独新增 [`docs/ai-testing-acceptance.md`](ai-testing-acceptance.md)，核心补充：

1. `P0` **需求→测试可追溯矩阵**：每条需求绑定机器可判定的验收用例 ID，未绑定即视为未完成。
2. `P0` **确定性/密封测试台**：单命令拉起全套依赖 + 注入时钟/随机种子/固定 UUID，AI 可无人值守可重复运行。
3. `P0` **防作弊控制**：变异测试（mutation testing）分数门禁、property-based 不变量测试、Golden 文件人类所有权、断言弱化检测——防止“既写代码又写测试”的 AI 把测试写松或改绿。
4. `P0` **AI 验收闭环**：红→绿→全量 `make verify`→自评清单→变异/覆盖→产出机器可读验收报告（含 REQ 覆盖、性能数字、Trace/账本断言）作为 PR 证据。
5. `P1` **非功能自动门禁**：性能阈值、峰值内存（100 并发大图）、goroutine 泄漏、连接池耗尽作为**会让 CI 失败的测试**，而非人工观察。

详见该文档。

## 11. 事件 / 钩子 / 插件机制（本次重构新增强诉求）

本次重构必须内建三类扩展点，为插件式开发铺路，详见 [`docs/plugin-architecture.md`](plugin-architecture.md)：

- **钩子（Hook，同步在途）**：绑定流水线阶段（after_parse/before_route/before_upstream/after_upstream/before_response 及生命周期），可读写请求上下文、可短路拒绝；受延迟预算与超时/panic 隔离约束；无钩子时零开销快速通道。
- **事件（Event，异步旁路）**：生命周期节点发布类型化事件，订阅者做审计、指标、Trace 导出、webhook、异步结算；发布**永不阻塞数据平面**，队列满按策略降级；资金类事件用 outbox 保证不丢。
- **插件（Plugin）**：受静态单二进制约束，首选**编译期注册表**（零热路径开销，泛化自现有 `Adaptor`）；可选 WASM（wazero/extism）承载不可信在途逻辑、gRPC 承载重型旁路插件。插件声明能力权限、配置 schema、资源上限，纳入不可变配置快照并按请求记录激活版本。
