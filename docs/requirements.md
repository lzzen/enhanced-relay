# 需求与范围

> 需求编号：本文每条 `P0`/`P1` 需求应分配稳定 ID（如 `REQ-TRACE-001`），并在测试中通过 `req.Covers(...)` 绑定，形成机器可判定的可追溯矩阵。详见 `docs/ai-testing-acceptance.md`。

## 1. 建设目标

构建一个独立的 AI 增强网关，对请求进入现有网关前、现有网关访问实际上游时，以及响应返回客户端前提供可观测和可编程能力。

首要目标不是替代现有系统的全部功能，而是建立以下闭环：

```text
可观测 → 可定位 → 可治理 → 可计费 → 可自动验证
```

## 2. 功能需求

### 2.1 请求追踪

- 为每次请求生成全局唯一 `trace_id`。
- 记录请求上传、规则处理、图片处理、排队、路由、建连、上游首字节、上游下载、OSS 上传及客户端写出耗时。
- 记录原始模型、路由模型、实际上游模型、原始接口和实际上游接口。
- 记录每次渠道尝试、网络出口、失败原因和耗时。
- 支持按 trace、用户、模型、渠道、状态和耗时查询。

### 2.2 请求审计

- 默认只保存参数名称、类型、长度、数量以及安全枚举值。
- prompt、messages、图片、音频、文件、Authorization 和 Cookie 默认不保存原文。
- 支持针对特定 trace、用户、模型或时间窗口临时开启受控调试快照。
- 调试快照必须加密、自动过期、权限隔离并记录访问审计。
- 展示原始请求、规则修改结果、实际上游请求之间的脱敏差异。

### 2.3 参数规则

- 支持 JSON、query、headers、form-urlencoded 和 multipart/form-data。
- 支持 set、delete、rename、copy、默认值、正则替换、数组过滤和字段映射。
- 支持字符串、数值、布尔、存在性、宽高和总像素条件。
- 支持 AND、OR、NOT、allowlist、denylist、reject、warn-only 和 shadow。
- 支持模型、接口路径和协议转换规则。
- 规则必须版本化，支持校验、影子、灰度、发布和回滚。

### 2.4 图片处理

- 识别真实格式和尺寸，限制输入文件大小、解码像素及图片数量。
- 修正 EXIF 方向，按规则缩放、压缩或转换格式。
- 默认不放大；透明图片不得无提示转换成丢失透明通道的格式。
- 删除不必要的 EXIF 元数据。
- 图片处理必须使用独立有界 worker pool。

### 2.5 响应转存 OSS

- 支持 Base64、Data URL 和上游临时 URL。
- 将结果流式上传到 OSS/S3，并返回公开 URL 或短期签名 URL。
- 支持私有 Bucket、生命周期、CDN、内容哈希和对象命名策略。
- OSS 失败支持 strict、fallback_base64、fallback_upstream_url 三种策略。

### 2.6 路由和网络出口

- 支持国内、海外及供应商专用出口。
- 按优先级、权重、成功率、P95 延迟、并发、成本、地域和能力标签选择。
- 支持熔断、半开恢复、限流、并发限制和回退链。
- 生图创建请求不得在无法确认上游未受理时自动重试。

### 2.7 异步任务

- 对接同步和异步上游生图接口。
- 对外提供创建、查询、取消及可控重试接口。
- 维护本地任务 ID 与上游任务 ID 映射。
- 支持 webhook、轮询、幂等、重启恢复、超时和结果持久化。

### 2.8 计费

- 分为估价、额度预留和最终结算。
- 支持模型、渠道、分组、尺寸、质量、数量、输入/输出图片、实际结果及网络出口维度。
- 每个请求冻结计费规则版本和路由决策。
- 使用整数最小货币单位，支持退款、补扣、幂等结算和完整明细。

### 2.9 可扩展性（事件 / 钩子 / 插件）

- 提供**钩子机制**：在流水线关键阶段（解析、路由、上游、响应及生命周期）注入同步逻辑，可读写请求上下文并可短路拒绝；无钩子时零开销。
- 提供**事件机制**：在生命周期节点发布类型化事件供异步订阅（审计、指标、Trace、webhook、结算），发布永不阻塞数据平面。
- 提供**插件机制**：核心增强能力（规则、图片、OSS、路由、计费、上游适配器）以统一插件形式实现，支持编译期注册；插件声明能力权限、配置 schema 与资源上限，纳入配置快照并按请求记录激活版本。
- 任一插件/钩子可一键禁用并回退透明代理路径。
- 扩展点必须可观测、受延迟预算与失败隔离约束。详见 `docs/plugin-architecture.md`。

## 3. 非功能需求

- 普通透明代理额外 P95 延迟目标低于 5 ms。
- Trace 覆盖率 100%，关键阶段数据完整率不低于 99.9%。
- 日志、Trace 或控制平面故障不得默认阻断正常请求。
- 所有队列、Body、图片像素、并发和重试均有硬上限。
- 数据平面不逐请求查询数据库；配置使用内存不可变快照。
- 客户端取消必须向下游传播。
- 支持多实例和滚动升级；滚动升级需支持连接排空（drain），收敛在途 SSE、异步提交与 OSS 上传。

## 3.1 平台与安全需求（补充）

- 配置以版本化不可变快照下发，具备 schema 校验、原子切换、坏配置自动回滚与启动 fail-fast。
- 敏感密钥（上游 API Key、OSS 凭证、调试快照密钥、内部 mTLS 证书）加密存储，支持轮换与最小权限。
- 内部跳段（现有网关 → Egress）必须鉴权（内部 mTLS 或签名短期令牌）并做 Provider 白名单校验，同时检测并拒绝代理循环。
- 数据库变更使用显式迁移工具与版本表，禁止生产环境依赖 AutoMigrate。
- 账本等资金相关数据具备明确备份与灾备目标（RPO/RTO）。
- 若存在多租户，限流、并发、配额与审计策略按租户隔离。

## 4. 第一阶段明确不做

- 不在首版完全替代现有网关的用户体系。
- 不在可观测性稳定前同时上线复杂动态路由和复杂计费。
- 不默认保存完整用户请求或图片。
- 不对不确定是否已受理的生图请求进行自动跨渠道重试。
