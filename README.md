# AI Enhanced Gateway

面向 AI API 与生图场景的可观测、可编程增强网关设计仓库。

本仓库当前保存需求、技术架构、测试体系及实施计划，供正式开发前进行独立评审。尚未包含生产代码。

## 背景

现有网关链路缺少足够的请求透明度。曾出现客户端请求总耗时达到约 800 秒，而实际上游生图只耗时几十秒，却无法判断时间消耗在请求上传、网关内部、跨境网络、重试、响应处理、OSS 上传还是客户端回传阶段。

同时，实际业务需要在不依赖既有网关内部修改的情况下实现：

- 请求全链路追踪和脱敏审计；
- JSON、表单和 multipart 参数过滤、替换及拒绝；
- 参考图压缩与图片安全检查；
- Base64/临时 URL 结果转存 OSS，并向客户端返回 URL；
- 国内外网络出口及渠道智能选择；
- 渠道分组、能力约束、熔断与回退；
- 同步和异步上游的统一适配；
- 对外同步、异步生图接口；
- 按模型、尺寸、质量、数量及实际结果结算的计费引擎；
- 可持续运行的自动化测试和性能回归体系。

## 当前推荐结论

1. 建设独立增强网关，不依赖修改现有网关内部代码。
2. 最终采用入口与出口双观察点：客户端先经过增强网关，现有网关向供应商发出的请求也经过增强网关的 Egress。
3. 第一阶段优先完成 Trace、分段耗时和脱敏请求差异，先解决“发生问题但无法追溯”。
4. 数据平面使用 Go，支持 Linux 单二进制及 Docker 两种发布方式。
5. 图片处理使用有界资源池；首版优先纯 Go 实现，必要时再拆分 libvips 图片服务。
6. 所有线上事故、协议适配、规则与计费路径必须有自动化回归测试。

## 评审状态

当前状态：`Draft / Request for Review`

建议评审者重点检查：

- 双代理拓扑是否存在循环、认证泄露或请求关联缺口；
- 生图请求在模糊失败时的幂等与重复扣费风险；
- 计费所有权与现有系统之间的边界；
- 图片处理峰值内存、OSS 故障降级和 SSRF 防护；
- 自动化测试是否覆盖 800 秒慢请求类故障；
- 第一阶段范围是否足够小，能快速上线并产生诊断价值。
